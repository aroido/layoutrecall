# v2 Release Designer Audit

Status: recommendation-only audit from the designer lane. None of the items below are auto-approved for implementation; PM must explicitly accept, defer, or reject them.

## Review scope

I reviewed the current menu bar and settings experience with a macOS-native lens, focusing on the v2.0.1 brief areas: onboarding, empty states, trust signals, and recovery controls.

Reviewed artifacts:

- generated UI snapshots from `swift test --filter renderMenuAndSettingsSnapshots`
- `Sources/LayoutRecallApp/MenuContentView.swift`
- `Sources/LayoutRecallApp/SettingsView.swift`
- `Sources/LayoutRecallApp/AppPresentation.swift`
- localized user-facing copy in `Sources/LayoutRecallKit/Resources/en.lproj/Localizable.strings`

## Surface-fit verdict

The product is using the right overall surface: LayoutRecall belongs in the menu bar because it exposes glanceable status and fast recovery actions. However, the healthy-state menu is still acting too much like a tiny dashboard instead of a low-attention utility surface. Deeper education, setup, and diagnostics should continue to live in Settings.

## Prioritized findings

### 1. High — first-run onboarding is still implicit instead of guided

**What is wrong**

First-run users are expected to infer the setup sequence from scattered hints. When no profile exists, the menu and restore pane mainly push `Save First Profile`, but they do not explain the full trust chain: save a baseline, install `displayplacer`, decide whether automatic restore should run, and optionally enable launch at login.

**Why it matters on macOS**

Menu bar utilities earn trust by being quiet **after** setup, not by being mysterious **during** setup. Asking users to piece together background behavior from several panes increases anxiety about what will run automatically.

**Evidence**

- The no-profile flow maps directly to a single `saveNewProfile` primary action with no checklist or guided setup state: `Sources/LayoutRecallApp/AppPresentation.swift:469-592`.
- The restore pane only shows one-line hints and action buttons, not a step-by-step setup model: `Sources/LayoutRecallApp/SettingsView.swift:173-258`.
- Launch-at-login status is isolated in General instead of appearing where automatic restore is configured: `Sources/LayoutRecallApp/SettingsView.swift:877-991`.
- Current copy confirms the flow is fragmented: `Sources/LayoutRecallKit/Resources/en.lproj/Localizable.strings:109-121`, `313-318`.

**Shortest plausible fix**

Add a dedicated "Get automatic restore ready" setup card for the no-profile / dependency-missing / app-auto-restore-off states. Show a 3-4 step checklist:

1. Save a profile
2. Install restore tool
3. Turn on automatic restore
4. Optional: enable launch at login

Each step should show clear status, one CTA, and one short explanation of why it matters.

### 2. High — the healthy-state menu still asks for too much attention

**What is wrong**

Even in the healthy state, the menu opens as a tall stack with a full status card, a second quick-controls card, inline actions, and a nested `More` menu. This reads like a compact control panel, not a background utility that should confirm state at a glance.

**Why it matters on macOS**

Menu bar surfaces should answer the top question quickly and then get out of the way. A calm healthy state should bias toward one concise status summary plus one or two obvious actions.

**Evidence**

- The menu always renders a large status block, optional primary action, quick controls, and footer inside a fixed-width panel: `Sources/LayoutRecallApp/MenuContentView.swift:20-47`.
- The healthy-state snapshot generated by `renderMenuAndSettingsSnapshots()` still consumes a full card stack instead of collapsing to a glanceable summary: `Tests/LayoutRecallAppTests/ViewSnapshotSmokeTests.swift:8-79`.
- Secondary actions are split between inline buttons and a generic `More` menu: `Sources/LayoutRecallApp/MenuContentView.swift:260-369`.

**Shortest plausible fix**

For the healthy state only, collapse the menu to:

- one status row (`Stable · Office Dock`)
- one light secondary line (`Auto restore on · 2 displays · last verified <time>`)
- one prominent recovery action (`Restore Now`) only when it is genuinely useful
- a compact overflow menu for profile save / swap / settings

Keep the richer card layout only for warning and recovery-needed states.

### 3. High — recovery choices are not explicit enough when trust is low or the user intentionally changed the layout

**What is wrong**

Low-confidence and manual-layout-override states do not present a clear, user-centered decision model. The app mainly offers `Restore Now`, `Save Profile`, or a buried swap action, but it does not clearly say: "restore the saved layout," "keep what I changed," or "review before restoring."

**Why it matters on macOS**

Recovery controls need explicit intent labels, especially for a utility that can move displays in the background. Ambiguous actions create fear of accidental restores.

**Evidence**

- `manualLayoutOverride` has no primary action at all, while low-confidence defaults to `fixNow`: `Sources/LayoutRecallApp/AppPresentation.swift:508-568`.
- The restore pane shows context-specific hints, but the available actions do not map cleanly to those hints: `Sources/LayoutRecallApp/AppPresentation.swift:572-592`, `Sources/LayoutRecallApp/SettingsView.swift:209-253`.
- `Swap Positions` is treated as a generic secondary action rather than a clearly risk-labeled recovery option: `Sources/LayoutRecallApp/MenuContentView.swift:315-337`, `Sources/LayoutRecallApp/SettingsView.swift:242-253`.

**Shortest plausible fix**

In low-confidence, manual-recovery, and manual-layout-override states, replace the generic action cluster with an explicit recovery group:

- **Restore saved layout**
- **Keep current layout**
- **Save current layout as new profile**
- **Swap side displays**

Support this with one-line microcopy that explains which choice is safest right now.

### 4. Medium — empty states are accurate but fragmented across panes

**What is wrong**

Profiles, diagnostics, and restore each have their own empty-state copy, but they do not feel like one coherent story. The user must move between panes to understand what to do next.

**Why it matters on macOS**

A desktop utility should minimize hunt-and-peck navigation. Empty states should keep the user in one lane until they finish setup.

**Evidence**

- Profiles has a save card and a separate empty box: `Sources/LayoutRecallApp/SettingsView.swift:398-433`.
- Diagnostics simply says no history exists yet, with no suggestion about what event will create proof: `Sources/LayoutRecallApp/SettingsView.swift:819-873`.
- The relevant copy exists, but it is distributed across several strings with no shared setup framing: `Sources/LayoutRecallKit/Resources/en.lproj/Localizable.strings:109-127`, `183-186`.

**Shortest plausible fix**

Use a shared empty-state pattern across Restore, Profiles, and Diagnostics:

- short explanation
- what will happen next
- one primary CTA
- one trust line (`Nothing runs automatically until a saved profile and restore tool are ready.`)

### 5. Medium — trust signals exist, but the app does not explain background behavior at the moment of decision

**What is wrong**

The UI shows good status badges (`Restore tool ready`, `Launch at login`, confidence labels), but it still expects users to translate those signals into operational trust on their own.

**Why it matters on macOS**

Background apps need to explain what they monitor, when they act, and what proof the user can inspect afterward. People should not have to open Diagnostics to confirm whether the app acted safely.

**Evidence**

- Trust-relevant badges are visible in Restore and General, but launch-at-login approval messaging is only available as a status string: `Sources/LayoutRecallApp/SettingsView.swift:290-367`, `877-991`.
- A `launchAtLogin.requiresApproval` message exists but is not surfaced in the restore setup lane: `Sources/LayoutRecallKit/Resources/en.lproj/Localizable.strings:313-318`.
- Diagnostics already contains useful proof, but it is not echoed back into the restore overview: `Sources/LayoutRecallApp/SettingsView.swift:756-873`.

**Shortest plausible fix**

Add a compact trust explainer in the restore/setup area:

- what LayoutRecall watches for
- what must be true before automatic restore runs
- where to inspect the last verified action

Also surface the latest successful or failed restore summary directly in the Restore pane so the user gets proof without navigating away.

## Recommended PM calls

### Strong v2.0.1 candidates

These align directly with the release brief and look high-value / low-risk:

1. Add a setup checklist card for first-run and degraded states.
2. Collapse the healthy-state menu into a calmer glanceable summary.
3. Rewrite low-confidence and manual-override recovery controls into explicit user choices.
4. Add one trust/proof summary row to the Restore pane (`Last verified restore`, `Last blocked restore`, etc.).

### Better for v2.0.2 unless PM wants them now

These are worthwhile, but less urgent than the clarity/trust fixes above:

- richer diagnostics export or copy actions
- profile-management bulk tools
- keyboard-first setup flows for advanced users
- user-tunable menu-bar visibility / noise preferences beyond current launch-at-login and update controls

## Suggested acceptance note for PM

If PM wants to keep v2.0.1 tightly scoped, I recommend accepting the first four v2.0.1 candidates above and deferring broader customization work. They improve trust and clarity without changing the core restore engine or widening product scope.
